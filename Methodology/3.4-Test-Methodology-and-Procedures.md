# 3.4 Test Methodology and Procedures

Testing is necessary to verify that LYDO Connect works correctly, protects data integrity, supports its intended users, and meets the quality expectations of the study. Because the platform includes public pages, administrative modules, and database-backed workflows, the testing methodology must cover more than simple page navigation.

## 3.4.1 Testing Strategy and Technique

The study will primarily use **black-box testing**, **integration testing**, **system testing**, and **user-oriented evaluation**. Black-box testing is appropriate because the goal is to validate whether each module behaves correctly based on its expected inputs and outputs. Integration testing is needed because many workflows connect frontend forms, database functions, storage, and role-based access rules. System testing is needed because the project is deployed as one integrated platform. User-oriented evaluation is needed because the portal must remain understandable to youth users, citizens, and administrators.

## 3.4.2 Functional Test Procedures

Functional testing will verify whether each major module performs its intended task. The following areas should be tested:

| Module | Sample Functional Checks |
|---|---|
| Authentication | account registration, sign in, sign out, role-based redirection |
| Programs and events | listing display, detail page loading, record filtering, source-post reference display |
| Registration | required field validation, successful registration, cancellation, profile update, status reflection |
| Organizations | "View Organization Info" action, full detail rendering, official source-link visibility, and null-safe display for missing fields |
| Transparency documents | upload, metadata validation, filtering, public retrieval |
| Compliance and financial records | create, update, save, display, and summary calculation |
| Citizen desk | ticket creation, auto reference generation, tracking, status update |
| Users and roles | role assignment, profile update, access restriction |
| Audit logs | automatic logging of create, update, and delete actions |

## 3.4.3 Usability Test Procedures

Usability testing will determine whether the interface is understandable and practical for intended users. This will include:

- checking whether menus, labels, buttons, and forms are easy to understand;
- checking whether common tasks can be completed without confusion;
- checking whether pages remain readable on desktop and mobile devices;
- checking whether public information is easy to locate through clear navigation;
- collecting evaluator comments about layout, clarity, consistency, and ease of use.

## 3.4.4 Performance, Compatibility, and Reliability Procedures

Performance and reliability testing will focus on whether the platform remains responsive and stable during normal use.

| Area | Procedure |
|---|---|
| Performance | Measure whether common page loads, table rendering, and dashboard fetches complete in acceptable time under normal connectivity |
| Compatibility | Open the system in current desktop and mobile browsers and verify layout, navigation, and form behavior |
| Reliability | Repeat create, update, cancel, and track operations to ensure records remain consistent |
| Record consistency | Verify that registration, ticket, document, and compliance updates remain correct after repeated operations |

## 3.4.5 Security and Data Integrity Procedures

Security and data integrity testing will verify whether protected actions remain restricted and whether invalid data are rejected.

| Focus Area | Procedure |
|---|---|
| Authentication | Verify that protected modules cannot be used without valid authentication |
| Authorization | Verify that admin-only pages and actions are blocked for unauthorized users |
| Input validation | Test invalid emails, invalid contact numbers, missing fields, and invalid dates or amounts |
| Source validation | Confirm that organization records show official source links and that invalid, malformed, or non-credible source links are rejected according to configured validation rules |
| Null-safe handling | Confirm that optional organization fields with unavailable verified data are stored as null and rendered safely without crashes or misleading placeholder claims |
| Database constraints | Confirm that Supabase schema constraints prevent impossible values, duplicate active records, and invalid source-reference states for organization records |
| Auditability | Confirm that admin-managed changes create audit log entries |

## 3.4.6 Defect Tracking and Acceptance Criteria

Defects discovered during testing will be recorded according to severity:

| Severity | Meaning | Action |
|---|---|---|
| Critical | Prevents core operation or risks data loss | Must be fixed before final evaluation |
| Major | Causes incorrect behavior in a key module | Must be fixed or justified before acceptance |
| Minor | Small display or workflow issue with workaround | Should be fixed if time allows |
| Cosmetic | Visual inconsistency only | May be deferred if it does not affect function |

The system will be considered acceptable for evaluation when:

- core public and admin workflows execute successfully;
- required data are stored and retrieved correctly;
- protected routes respect role restrictions;
- organization detail pages or modals correctly display official references and null-safe optional fields;
- no unresolved critical defects remain; and
- evaluators can complete representative tasks using the system.

## Summary

This testing methodology is broader and more appropriate for LYDO Connect than the template's limited web store testing examples because it includes participation, transparency, citizen ticketing, data integrity, and admin accountability workflows.
