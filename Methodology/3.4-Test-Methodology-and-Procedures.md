# 3.4 Test Methodology and Procedures

Testing is an essential part of the development of LYDO Connect because the system handles public information, youth program and event registration, youth service requests, transparency records, financial disclosure, policy agreement enforcement, and administrative governance records. The purpose of the test methodology is to verify that each module functions correctly, integrates properly with other parts of the system, protects sensitive data, and provides a usable experience for youth users, admin and staff personnel, and evaluators.

## 3.4.1 Unit Testing

Unit testing will be conducted to verify the correctness of individual functions, components, and modules of LYDO Connect. These tests will focus on smaller units such as user registration forms, login validation, policy agreement handling, program and event forms, registration validation, youth desk ticket creation, organization information display, transparency record forms, financial data calculations, role handling, and audit-related utilities.

Each unit will be tested in isolation to confirm that it accepts valid input, rejects invalid input, produces the expected output, and handles errors properly. For example, tests may verify that Philippine mobile numbers are validated correctly, required form fields display proper error messages, registration status values are handled consistently, and optional organization fields are rendered safely when verified source data are unavailable.

To support efficient unit testing, the developers may use available testing frameworks and libraries such as Vitest and React Testing Library. These tools allow the definition and execution of repeatable test cases, making it easier to identify errors early and maintain the quality of the codebase as the system grows.

## 3.4.2 Integration Testing

Integration testing will evaluate how the different modules of LYDO Connect work together. Since the system connects the public portal, admin portal, authentication, Supabase database records, storage, policy agreement records, registration monitoring, and external registration synchronization support, it is important to verify that data are exchanged correctly between related components.

The integration tests will include checking the connection between user accounts and profiles, program/event records and registration forms, registration records and participation history, youth desk submission and tracking, transparency document management and public display, financial records and dashboard summaries, and user or role changes and audit logs. The interaction between the portal registration records and the Google Forms/Sheets synchronization workflow will also be examined when external attendance integration is enabled.

These tests will verify data flow, error handling, boundary cases, and exception scenarios. Examples include failed registration sync attempts, missing source references, invalid document links, restricted admin access, and records that are saved successfully in one module but must appear correctly in another module.

## 3.4.3 System Testing

System testing will evaluate LYDO Connect as a complete and integrated application. This phase will test whether the full system satisfies the documented functional and non-functional requirements. The system will be tested through realistic scenarios that represent actual usage by youth users, admin personnel, and staff personnel.

The test cases will cover end-to-end workflows such as signing up, signing in, accepting the active Terms and Privacy Policy, browsing programs and events, registering for a program or event, viewing participation history, opening organization information, inspecting official source references, submitting a youth desk ticket, tracking a ticket, viewing transparency documents, checking financial disclosure data, and using the barangay map.

For the admin interface, system testing will cover dashboard viewing, managing programs, managing events, monitoring registrations, previewing external sheets, exporting CSV files, managing organizations, maintaining barangay map data, uploading or linking transparency documents, updating transparency board records, managing financial DSS records, handling youth desk tickets, reviewing audit logs, managing users, and managing roles and permissions.

The goal of system testing is to confirm that the application behaves consistently under normal usage, maintains data integrity, and supports the expected workflows before final evaluation or deployment.

## 3.4.4 Acceptance Testing

Acceptance testing will determine whether LYDO Connect meets the needs and expectations of its intended users and stakeholders. This testing phase provides youth users, admin and staff personnel, and evaluators an opportunity to assess whether the system is understandable, useful, and aligned with the objectives of the study.

During this phase, selected participants will perform representative tasks based on their roles. Youth users may be asked to browse programs and events, register for an activity, view their profile, access organization information, inspect source references, submit a youth desk request, and view transparency records. Admin and staff personnel may be asked to create or update records, monitor registrations, review sync status, manage transparency and financial data, handle youth tickets, and inspect audit logs.

Feedback collection methods such as structured questionnaires, interviews, and open-ended comments may be used to gather insights from participants. The collected feedback will be analyzed to identify usability concerns, missing requirements, workflow issues, and possible enhancements. The system may be considered acceptable if the major workflows are completed successfully and the evaluation results fall within the acceptable or highly acceptable range.

## 3.4.5 Performance Testing

Performance testing will assess whether LYDO Connect can provide a smooth and responsive experience during normal and expected usage. Since the system includes public browsing, registration submission, dashboard summaries, table filtering, CSV export, external sheet preview, financial records, transparency records, and youth desk tracking, it must remain responsive while handling common youth user, admin, and staff tasks.

Load testing may be performed to determine how the system behaves when multiple users access public pages, submit forms, or view database-backed records. Stress testing may also be conducted to observe how the system responds when the amount of data or number of simultaneous actions increases beyond normal expectations.

Performance checks may include page loading time, search and filter responsiveness, registration submission speed, admin table loading time, dashboard data retrieval, CSV export generation, map rendering, and the behavior of registration sync-related actions. Any identified performance bottlenecks will be reviewed and optimized to improve responsiveness, stability, and scalability.

## 3.4.6 Security Testing

Security testing will be conducted to protect user information, administrative records, youth service requests, transparency records, and governance data. Since LYDO Connect includes authenticated accounts and role-based administrative features, security testing is necessary to ensure that only authorized users can access protected workflows.

The security tests will include checking authentication behavior, role-based redirection, admin route protection, sign-out behavior, policy agreement enforcement for authenticated non-admin users, and restrictions on admin-only actions. Input validation will also be tested using invalid emails, invalid contact numbers, missing required fields, invalid dates, invalid amounts, malformed links, and improper registration or sync values.

Security testing will also examine database constraints, audit logging, and data integrity rules. The system should prevent duplicate or impossible records, protect administrative actions, and record important create, update, and delete operations in the audit logs. Vulnerability scanning, code review, and penetration-style checks may be performed when available to identify weaknesses before deployment.

## 3.4.7 Usability Testing

Usability testing will evaluate how easily intended users can understand and use LYDO Connect. The testing will observe users as they interact with the public portal and admin portal, focusing on whether navigation, labels, buttons, forms, tables, and feedback messages are clear and consistent.

For the public portal, usability tasks may include finding a program or event, opening a detail page, registering for an activity, viewing profile participation history, checking organization information, opening source references, submitting a youth desk ticket, tracking a ticket, and finding transparency or financial records. For the admin portal, usability tasks may include using the dashboard, managing program/event records, filtering registrations, exporting CSV data, previewing external sheets, updating youth tickets, and reviewing audit logs.

The main objective of usability testing is to identify user experience issues, confusing workflows, unclear labels, difficult forms, or layout problems. User feedback will help improve the interface so that the system becomes more intuitive, efficient, readable, and practical for the intended users.

## 3.4.8 Compatibility Testing

Compatibility testing will verify whether LYDO Connect works properly across the platforms and devices commonly used by the target users. The system will be tested on current desktop and mobile browsers to confirm that pages, forms, navigation menus, tables, maps, modals, and public records remain usable and readable across different screen sizes.

Testing will include checking browser compatibility, responsive layout behavior, mobile navigation, form controls, map display, file links, embedded source references, and admin table usability. The goal is to ensure that youth users, admin and staff personnel, and evaluators can access the system consistently whether they are using a desktop computer, laptop, tablet, or mobile phone.

Any compatibility issues discovered during testing will be documented and addressed to improve accessibility, reliability, and consistency of the user experience across supported environments.
