# 3.3.1 Process Model

The study adopts a hybrid **Design Science Research (DSR)** and **Rapid Application Development (RAD)** process model that matches the current LYDO Connect workflow.

**DSR as the research framework** guides the identification of the operational problem in youth-organization compliance and monitoring, the definition of solution objectives, the development of the web-based artifact, the demonstration of the system in its intended LYDO context, and the evaluation of its functional usefulness.

**RAD as the development strategy** supports short iterative cycles of design, prototyping, validation, revision, and release. This is appropriate because the system was refined through repeated updates to its role-based routing, forms, review states, notifications, and Supabase-backed records.

This combined process model is appropriate for LYDO Connect because the current system is composed of interconnected modules that evolved incrementally around the actual site workflow. These modules include authentication and policy agreement, organization profile management, compliance document submission and review, budget request processing, liquidation reporting, YPOP and YORP-related monitoring, transparency and news publishing, notifications, and activity-log-based admin monitoring.

In operational terms, LYDO Connect follows an iterative workflow:

1. Define the current workflow scope.
2. Design the module or change against the live workflow.
3. Implement the feature in the codebase and Supabase schema when needed.
4. Test the feature locally.
5. Revise the workflow based on issues found.
6. Commit or deploy the updated artifact.

This cycle is repeated across the current app workflow, especially for authentication, policy acceptance, organization profiles, document submission, budget and liquidation processing, transparency and news content management, notifications, YPOP and YORP monitoring, and admin-side review and audit tracking.
