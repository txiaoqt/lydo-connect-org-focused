# 3.0 Methodology

This section presents the methodology for developing **LYDO Connect** based only on the current application workflow in the repository. The implemented workflow covers sign-up, sign-in, sign-out, policy agreement gating, organization profile management, document submission, budget request CRUD with attachments, liquidation reporting after budget approval, budget allocation monitoring by barangay and district, news release previewing, public transparency posting, notifications, required document type management, and admin review.

## Research Design

The study uses a **descriptive-developmental research design**. It is descriptive because it identifies the needs, records, and operational tasks present in the current workflow. It is developmental because it focuses on designing and refining a working web application that supports the implemented LYDO Connect modules in the repository.

The study is guided by **Design Science Research (DSR)** and **Rapid Application Development (RAD)**. DSR provides the research structure for defining the problem, specifying the solution, building the artifact, and evaluating its usefulness. RAD supports iterative implementation, rapid feedback, and continuous improvement of the existing workflow behavior.

## SDLC Model

The study follows a **modified System Development Life Cycle (SDLC)** using the RAD approach. This fits the current application because the system is composed of multiple interconnected modules that require frequent refinement. The development cycle covers requirements analysis, interface design, database and API integration, prototype implementation, testing, revision, and evaluation for the implemented modules only.

## Data-Gathering Methods

Primary data are gathered through consultation, interview, and evaluation feedback from intended users, administrators, and qualified evaluators. These inputs define the functional needs of the system and validate whether the implemented workflow matches the operational requirements already reflected in the app.

Secondary data are gathered through document review of policy references, local office records, transparency requirements, and system-related materials that support the current app workflow. These sources inform the records, fields, and content types already supported by the application.

## System Development Approach

The study combines **DSR as the research framework** and **RAD as the software development strategy**. The artifact is built incrementally, verified through testing, and revised based on feedback from the implemented workflow.

For LYDO Connect, the current workflow supports:
- sign-up, sign-in, and sign-out
- policy agreement gating for non-admin users
- organization profile management
- document submission with uploaded files and preview
- budget request CRUD, file attachments, and released budget monitoring
- liquidation access after budget approval
- news release cards with Facebook preview pages
- public transparency posts
- notifications
- required document type management
- admin monitoring, review, audit-friendly content updates, and barangay allocation summaries

## Summary

Overall, the methodology is anchored on a descriptive-developmental design, a modified SDLC using RAD, and iterative validation of the current app workflow. This approach is appropriate because it supports the development and review of the actual modules already implemented in LYDO Connect, including barangay-level budget monitoring, and it excludes features that are not part of the current repository workflow.
