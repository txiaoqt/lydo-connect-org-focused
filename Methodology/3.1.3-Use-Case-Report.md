# 3.1.3 Use Case Report

This section summarizes the core use cases in LYDO Connect based on the current application workflow. The report follows the same format used for the diagram and stays within the implemented modules only.

## Table 1. Use Case Report of Sign Up and Sign In

| Field | Details |
|---|---|
| USE CASE ID | UC-01 |
| USE CASE NAME | Sign Up and Sign In |
| SCENARIO | A public user creates an account or signs in to access LYDO Connect services. |
| TRIGGERING EVENT | The public user selects the sign-up or sign-in option. |
| BRIEF DESCRIPTION | This use case allows a public user to create an account or authenticate an existing account before entering the role-based workflow. Non-admin users are routed through the policy agreement gate before using protected features. |
| ACTORS | Public Visitor, Organization User |
| INCLUDE USE CASE | Authenticate User; Load Active Policy |
| EXTEND USE CASE | Login Error; Policy Agreement |
| PRE-CONDITIONS | The user has access to the sign-up or sign-in page and has the required account details. |
| POST-CONDITIONS | The user account is authenticated and the user may proceed to the appropriate portal. |

### Flow of Events

| ACTOR | SYSTEM |
|---|---|
| 1. The public user opens the sign-up or sign-in page. | 1. The system displays the account form. |
| 2. The public user enters the required account details. | 2. The system validates the fields and credentials. |
| 3. The public user submits the form. | 3. The system authenticates the user account. |
| 4. The public user continues to the portal. | 4. The system loads the policy gate for non-admin users or opens the admin portal for admin users. |

| ALTERNATIVE FLOW | If validation or authentication fails, the system displays an error and returns the user to the form. |
|---|---|
| EXCEPTION CONDITIONS | Invalid credentials, missing required fields, or authentication service error may prevent access. |

## Table 10. Use Case Report of Monitor Barangay Budget Allocation

| Field | Details |
|---|---|
| USE CASE ID | UC-10 |
| USE CASE NAME | Monitor Barangay Budget Allocation |
| SCENARIO | The LYDO Admin reviews released budget totals by barangay and district to support monitoring. |
| TRIGGERING EVENT | The admin opens the budget monitoring page and selects the barangay allocation tab. |
| BRIEF DESCRIPTION | This use case allows the admin to review released and approved budget amounts grouped by barangay and district, then filter the results for monitoring and reporting. |
| ACTORS | LYDO Admin |
| INCLUDE USE CASE | Load Budget Summary; Apply District Filter; Apply Barangay Filter |
| EXTEND USE CASE | Empty Result State; Filter Reset |
| PRE-CONDITIONS | Budget request records with released amounts already exist in the system. |
| POST-CONDITIONS | The admin can view filtered allocation summaries for monitoring. |

### Flow of Events

| ACTOR | SYSTEM |
|---|---|
| 1. The admin opens the budget monitoring page. | 1. The system displays the monitoring summary cards and allocation table. |
| 2. The admin selects the barangay allocation tab. | 2. The system loads released budget totals grouped by organization barangay and district. |
| 3. The admin chooses a district or barangay filter. | 3. The system filters the list and recalculates the summary values. |
| 4. The admin reviews the filtered allocation data. | 4. The system displays the released amount, approved amount, remaining amount, utilization rate, and organization count. |

| ALTERNATIVE FLOW | If no records match the selected filter, the system shows an empty state with zero totals. |
|---|---|
| EXCEPTION CONDITIONS | Missing budget data or an invalid filter selection may prevent the summary from loading correctly. |

## Summary

The current use cases focus on the exact modules implemented in the application: authentication, policy acceptance, organization profile management, document submission, budget request CRUD, liquidation reporting, budget allocation monitoring, news release previewing and CRUD, transparency posting, required document type management, notifications, and admin review. Modules not present in the current workflow are excluded from this report.
