# 3.1.3 Use Case Report

The use case report explains the major system functions in narrative form. For LYDO Connect, the report should focus on the most critical workflows instead of copying generic e-commerce scenarios from the template.

## UC-01 Account Registration and Sign In

| Field | Description |
|---|---|
| Use Case ID | UC-01 |
| Use Case Name | Account Registration and Sign In |
| Primary Actor | Youth user or citizen |
| Goal | Create an account and access authenticated portal services |
| Preconditions | The user has internet access and a valid email address |
| Trigger | The user selects Sign Up or Sign In |
| Main Flow | 1. The user opens the authentication page. 2. The user enters required credentials and profile details. 3. The system validates the input. 4. The system creates or authenticates the account. 5. The system initializes the user profile and applicable default role. |
| Alternate Flow | Invalid email, duplicate account, wrong password, or incomplete required fields |
| Postconditions | The user gains access to profile-based features and protected participation workflows |

## UC-02 Browse Youth Records

| Field | Description |
|---|---|
| Use Case ID | UC-02 |
| Use Case Name | Browse Youth Records |
| Primary Actor | Youth user, citizen, public viewer |
| Goal | View programs, events, organizations, and advisories |
| Preconditions | Public portal is available |
| Trigger | The actor opens a listing page |
| Main Flow | 1. The actor selects a module such as Programs or Events. 2. The system retrieves published records. 3. The system displays summaries, filters, and navigation options. 4. The actor selects a specific record to view full details. |
| Alternate Flow | No records are available or the data source is temporarily unavailable |
| Postconditions | The actor sees the current public records and may proceed to a detail page |

## UC-03 Register for Program or Event

| Field | Description |
|---|---|
| Use Case ID | UC-03 |
| Use Case Name | Register for Program or Event |
| Primary Actor | Authenticated youth user |
| Goal | Submit participation details for a program or event |
| Preconditions | The user is authenticated and the selected record is open for registration |
| Trigger | The user opens the Registration tab of a record page |
| Main Flow | 1. The user enters full name, email, contact number, municipality, and barangay. 2. The system validates the input. 3. The system checks record status and capacity rules when applicable. 4. The system stores the registration. 5. The system updates participation history and membership if the record is a program. 6. If external sync is enabled, the record is queued for Google Form submission. |
| Alternate Flow | The user is not signed in, the event is full, the record is closed, or the input is invalid |
| Postconditions | The registration is recorded and reflected in the user's profile |

## UC-09 Monitor Registrations and Retry Sync

| Field | Description |
|---|---|
| Use Case ID | UC-09 |
| Use Case Name | Monitor Registrations and Retry Sync |
| Primary Actor | LYDO administrator or staff |
| Goal | Review participation records and manage Google Form sync status |
| Preconditions | Registration records exist and admin access is active |
| Trigger | The actor opens the Registrations module in the admin portal |
| Main Flow | 1. The actor selects event or program records. 2. The system displays local registrations and sync statuses. 3. The actor reviews pending or failed rows. 4. The actor triggers retry for failed sync rows or runs the worker once. 5. The system refreshes statuses after processing. |
| Alternate Flow | Sync endpoint fails, Google Form mapping changes, or no external sheet is configured |
| Postconditions | Registration monitoring is updated and failed rows may be re-queued for synchronization |

## Summary

The use case report confirms that LYDO Connect is centered on public information access, user participation, transparency publication, and administrative accountability. These are the use cases that should guide the diagrams, database schema, testing procedures, and evaluation criteria for the study.
