# 3.1.3 Use Case Report

This section summarizes the core use cases in LYDO Connect based on the current application workflow.

## 1. Sign Up / Sign In / Policy Agreement

| Item | Description |
|---|---|
| Actor | Public Visitor / Organization User |
| Goal | Create an account, access the system, sign out when needed, and accept the active policy before using protected features |
| Preconditions | The user has access to the sign-up or sign-in page |
| Main Flow | The user creates an account or signs in, the system authenticates the account, and the policy gate appears for non-admin users until acceptance is saved |
| Alternate Flow | If validation fails, the system displays the related error message and returns the user to the form |

## 2. Manage Organization Profile

| Item | Description |
|---|---|
| Actor | Organization User |
| Goal | Create or update the organization profile linked to the account |
| Preconditions | The user is signed in |
| Main Flow | The user fills in the organization profile fields, saves the record, and the system stores the updated profile |
| Alternate Flow | If required fields are missing, the system blocks saving and shows validation feedback |

## 3. Submit Documents

| Item | Description |
|---|---|
| Actor | Organization User |
| Goal | Upload required document files for admin review |
| Preconditions | The organization profile exists and the user is signed in |
| Main Flow | The user selects a document type, uploads a file, reviews the preview, and submits the file for validation |
| Alternate Flow | If the file is invalid or missing, the system shows the error and prevents submission |

## 4. Manage Budget Request

| Item | Description |
|---|---|
| Actor | Organization User |
| Goal | Create, edit, save draft, submit, or delete a budget request |
| Preconditions | The organization profile exists |
| Main Flow | The user enters the activity title, description, proposed date, venue, requested amount, purpose/category, and supporting file, then saves or submits the record |
| Alternate Flow | If required details are incomplete, the system prevents saving or submission |

## 5. Submit Liquidation Report

| Item | Description |
|---|---|
| Actor | Organization User |
| Goal | Upload post-activity evidence for an approved budget request |
| Preconditions | The related budget request is approved |
| Main Flow | The user opens the liquidation record and uploads files such as photos or attendance sheets |
| Alternate Flow | If the budget is not yet approved, the liquidation record does not appear |

## 6. View News Release Preview

| Item | Description |
|---|---|
| Actor | Public Visitor / Organization User / LYDO Admin |
| Goal | Preview the Facebook source post before opening the external link |
| Preconditions | A news release record exists |
| Main Flow | The user opens the news release card, the system shows the preview page, and the source post can be opened in a new tab |
| Alternate Flow | If the source URL is unavailable, the system shows an unavailable preview state |

## 7. View Public Transparency Posts

| Item | Description |
|---|---|
| Actor | Public Visitor / Organization User / LYDO Admin |
| Goal | View published transparency posts |
| Preconditions | A transparency post record exists and is published |
| Main Flow | The user opens the transparency page, reviews the published record, and optionally opens the attached file when available |
| Alternate Flow | If the record is hidden or unavailable, the system shows no published content for that entry |

## 8. Manage News Releases

| Item | Description |
|---|---|
| Actor | LYDO Admin |
| Goal | Create, edit, publish, hide, preview, or delete a news release |
| Preconditions | The admin account is active |
| Main Flow | The admin enters the title, description, Facebook URL, posting date, and visibility, then saves the record |
| Alternate Flow | If validation fails, the system returns the admin to the form with errors |

## 9. Manage Required Document Types and Review Records

| Item | Description |
|---|---|
| Actor | LYDO Admin |
| Goal | Maintain required document type records and review organization submissions, budget requests, liquidations, transparency posts, notifications, and activity logs |
| Preconditions | The admin is signed in |
| Main Flow | The admin opens the relevant section, reviews the record, changes status where allowed, and the system stores the update and associated logs |

## Summary

The current use cases focus on the exact modules implemented in the application: authentication, policy acceptance, organization profile management, document submission, budget request CRUD, liquidation reporting, news release previewing and CRUD, transparency posting, required document type management, notifications, and admin review. Modules not present in the app workflow are excluded from this report.
