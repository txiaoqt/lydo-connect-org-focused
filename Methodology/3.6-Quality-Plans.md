# 3.6 Quality Plans

The quality plan defines how LYDO Connect will be checked for correctness, usability, reliability, and traceability.

## 3.6.1 Quality Standards and Objectives

The most relevant quality characteristics for the current app are:

| Quality Characteristic | Application to LYDO Connect |
|---|---|
| Functional suitability | The current modules should behave as described in the workflow |
| Usability | Mobile and desktop users should be able to complete tasks without confusion |
| Reliability | Records should remain consistent during create, update, and delete operations |
| Security | Protected routes and admin actions should remain role-based |
| Maintainability | Modules should remain modular and easy to update |
| Responsiveness | The app should remain usable on mobile screens and common browsers |
| Traceability | Admin changes should be visible through statuses, notifications, or logs |

## 3.6.2 Quality Assurance Activities

- review requirements before implementing a module
- verify the updated workflow against the current app screens
- test forms, dialogs, upload flows, and preview behavior
- confirm that validation rules reject incomplete or invalid records
- confirm that admin-only actions cannot be accessed by non-admin users
- review that current records and notifications remain traceable

## 3.6.3 Quality Control and Defect Management

| Activity | Control Measure |
|---|---|
| Functional checking | Execute defined test cases for each current module |
| Validation checking | Confirm required fields, status rules, and file handling behavior |
| Error tracking | Record issues by module and severity |
| Regression checking | Re-test impacted workflows after fixes |
| Acceptance review | Confirm no critical workflow issue remains before final use |

## 3.6.4 Backup, Recovery, and Change Management

The project should keep source code in version control, use controlled migrations for database changes, and retain uploaded files in managed storage. Important record corrections should be performed through the application workflow or reviewed migrations rather than unmanaged direct edits.

## Summary

The quality plan is limited to the modules in the current application workflow and avoids quality statements for features that are not implemented.
