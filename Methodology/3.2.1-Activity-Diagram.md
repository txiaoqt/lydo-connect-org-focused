# 3.2.1 Activity Diagram

Activity diagrams show how the major LYDO Connect workflows proceed from one action to the next. The diagrams below focus on the implemented workflows for policy agreement, portal-local registration, external registration sync support, citizen services, transparency publication, compliance records, organization source verification, and audit-supported administration.

## Figure 6.1. Activity Diagram of Policy Agreement Gate

```mermaid
flowchart TD
    A([Start]) --> B[Authenticated non-admin user opens the portal]
    B --> C[System loads the active policy version]
    C --> D{Policy found?}
    D -- No --> E[Show policy loading error with retry and sign-out actions]
    E --> C
    D -- Yes --> F[System checks user policy acceptance]
    F --> G{Already accepted active version?}
    G -- Yes --> H[Allow protected user workflow]
    G -- No --> I[Display Terms of Service and Privacy Policy modal]
    I --> J{User accepts?}
    J -- No --> K[Sign user out]
    J -- Yes --> L[Save user policy acceptance record]
    L --> M{Save successful?}
    M -- No --> I
    M -- Yes --> H
    H --> N([End])
    K --> N
```

## Figure 6.2. Activity Diagram of Program or Event Registration

```mermaid
flowchart TD
    A([Start]) --> B[Youth user opens a program or event record]
    B --> C{Is the user authenticated?}
    C -- No --> D[Prompt user to sign in or create an account]
    D --> B
    C -- Yes --> E{Has accepted active policy?}
    E -- No --> F[Show policy agreement modal]
    F --> E
    E -- Yes --> G[Display registration form]
    G --> H[User enters name, email, contact number, municipality, and barangay]
    H --> I[System validates required fields and PH mobile format]
    I -- Invalid --> J[Display validation message]
    J --> G
    I -- Valid --> K{Record type}
    K -- Event --> L[Check event status and capacity]
    K -- Program --> M[Check program publication status]
    L -- Closed or full --> N[Display registration unavailable message]
    M -- Not published --> N
    L -- Available --> O[Create or update event registration]
    M -- Published --> P[Create or update program registration]
    P --> Q[Create or reactivate program membership]
    O --> R{External attendance enabled?}
    Q --> R
    R -- Yes --> S[Mark sync status as pending]
    R -- No --> T[Mark sync status as skipped]
    S --> U[Display successful registration confirmation]
    T --> U
    U --> V[Admin or staff reviews local registration and sync status]
    V --> W([End])
    N --> W
```

## Figure 6.3. Activity Diagram of Registration Monitoring and External Sync

```mermaid
flowchart TD
    A([Start]) --> B[Admin opens Registrations module]
    B --> C[System loads events, programs, local registration rows, barangays, source, and sync status]
    C --> D[Admin selects event or program mode and applies filters]
    D --> E[System displays total rows, unique participants, pending sync, and table results]
    E --> F{Selected action}
    F -- Export --> G[Generate CSV from filtered rows]
    F -- Preview Sheet --> H[Normalize published Google Sheet URL and fetch CSV rows]
    F -- Retry Failed Row --> I[Queue selected registration by setting sync status to pending]
    F -- Run Sync Now --> J[Call protected sync-run-once endpoint with token]
    J --> K[Worker posts pending rows to Google Forms when integration is enabled]
    K --> L[Worker marks rows as synced, failed, or skipped]
    G --> M[Admin reviews result]
    H --> M
    I --> M
    L --> M
    M --> N([End])
```

```mermaid
    L --> M[Admin or staff reviews ticket queue]

    N --> O[Audit log records administrative change]

    P --> Q([End])
    K --> Q
```

## Figure 6.6. Activity Diagram of View Organization Info and Source Verification

```mermaid
flowchart TD
    A([Start]) --> B[User opens the Organizations page]
    B --> C[System loads published organization cards]
    C --> D{Organization records available?}
    D -- No --> E[Display empty state message]
    E --> F([End])
    D -- Yes --> G[User searches or filters organizations]
    G --> H[User clicks View Organization Info]
    H --> I[System loads full organization profile]
    I --> J[Display name, category, overview, mission, programs, projects, coverage area, and target beneficiaries]
    J --> K{Verified source reference available?}
    K -- No --> L[Display source unavailable note]
    K -- Yes --> M[User clicks Inspect Verified Source Reference]
    M --> N[System opens official source link in a new context]
    L --> O([End])
    N --> O
```

## Interpretation

- The policy workflow requires non-admin authenticated users to accept the active Terms of Service and Privacy Policy before protected user tasks continue.
- The registration workflow stores participation directly in `event_registrations` or `program_registrations`, with program registrations also maintaining `user_program_memberships` and sync status fields for external Google Forms processing.
- The registration monitoring workflow lets admins filter local rows, export CSV files, preview published Google Sheets, retry failed rows, and run the sync worker through the protected endpoint.
- The citizen desk workflow supports public tracking through `citizen_tickets` and administrative handling through assigned users, status updates, and audit logs.
- The transparency workflow shows how administrative records become public-facing governance information through disclosure, financial, compliance, and advisory modules.
- The organization information workflow is read-centric: users view organization details and inspect verified source references, without joining or membership state changes.
