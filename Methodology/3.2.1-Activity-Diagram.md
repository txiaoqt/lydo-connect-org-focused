# 3.2.1 Activity Diagram

Activity diagrams show how major workflows proceed from one action to the next. Since LYDO Connect supports several integrated services, the methodology should present the most critical operational flows instead of copying unrelated buying and selling processes from the template.

## A. Program or Event Registration and Sync Workflow

```mermaid
flowchart TD
    A[User opens program or event record] --> B{Is user authenticated?}
    B -- No --> C[Prompt user to sign in]
    B -- Yes --> D[User fills out registration form]
    D --> E[Client-side validation]
    E -- Invalid --> F[Display validation errors]
    E -- Valid --> G[Call registration RPC]
    G --> H{Record type}
    H -- Event --> I[Validate event status and capacity]
    H -- Program --> J[Validate program publication status]
    I --> K[Create or update event registration]
    J --> L[Create or update program registration]
    L --> M[Update program membership]
    K --> N{External attendance sync enabled?}
    M --> N
    N -- Yes --> O[Mark registration as pending for Google Form sync]
    N -- No --> P[Mark registration as skipped]
    O --> Q[Show successful registration to user]
    P --> Q
    Q --> R[Worker reads pending registrations]
    R --> S[Submit mapped values to Google Form]
    S --> T{Submission successful?}
    T -- Yes --> U[Mark row as synced]
    T -- No --> V[Mark row as failed and store error]
    V --> W[Admin retries failed sync]
    W --> R
```

## B. Citizen Desk Submission and Tracking Workflow

```mermaid
flowchart TD
    A[Citizen opens Citizen Desk page] --> B[Fill out ticket form]
    B --> C[Validate ticket fields]
    C -- Invalid --> D[Show input errors]
    C -- Valid --> E[Insert ticket record]
    E --> F[Database trigger generates reference number]
    F --> G[System returns reference number and status]
    G --> H[Citizen saves tracking details]
    H --> I[Citizen enters reference number and email for follow-up]
    I --> J[System executes tracking RPC]
    J --> K{Matching ticket found?}
    K -- Yes --> L[Display ticket status and timestamps]
    K -- No --> M[Show no match or invalid tracking details]
    L --> N[Admin reviews and updates ticket status]
    N --> O[Updated status becomes visible in tracking results and KPIs]
```

## C. Transparency Publication Workflow

```mermaid
flowchart TD
    A[Admin opens Documents or Transparency module] --> B[Enter metadata or upload file]
    B --> C[Validate file type, year, quarter, office, and barangay scope]
    C -- Invalid --> D[Display validation error]
    C -- Valid --> E[Save record to database or storage]
    E --> F[Audit trigger records the action]
    F --> G[Public portal fetches updated records]
    G --> H[User searches, filters, views, or downloads the published data]
```

## Interpretation

- The first activity diagram captures the system's most distinctive workflow because it joins portal registration, membership maintenance, and optional Google Form synchronization.
- The second diagram reflects the citizen service workflow that supports accountability and public responsiveness.
- The third diagram shows how administrative encoding becomes publicly accessible transparency information.
