# 3.1.1 Requirement Analysis

Requirement analysis identifies what LYDO Connect must do, who will use it, and what quality conditions the system must satisfy. For this study, requirement analysis is necessary because the system is not a simple event portal; it combines youth engagement services, transparency publication, financial disclosure, citizen service, and administrative governance in one web platform.

## Stakeholders and Their Needs

| Stakeholder | Primary Need | System Response |
|---|---|---|
| Youth users | Accessible information on programs, events, organizations, and participation services | Public portal with browse, registration, profile, and participation history modules |
| General citizens | Easy access to transparency records, advisories, financial data, and concern reporting | Transparency, citizen desk, service advisories, and financial disclosure modules |
| LYDO administrators | Centralized management of youth records and transparency content | Admin portal with CRUD, dashboards, compliance, and document management |
| SK and staff personnel | Operational monitoring and data updates | Registration monitoring, barangay data maintenance, and workflow dashboards |
| Evaluators and auditors | Traceable and accountable system activity | Audit logs, role-based access, validation rules, and standardized records |

## Functional Requirements

| ID | Requirement | Description |
|---|---|---|
| FR-01 | User account management | The system shall allow users to sign up, sign in, and sign out using authenticated accounts. |
| FR-02 | Role-based access control | The system shall separate public/user access from admin access based on assigned roles. |
| FR-03 | Program management | The system shall allow administrators to create, update, publish, and archive youth programs. |
| FR-04 | Event management | The system shall allow administrators to create, update, publish, and monitor events. |
| FR-05 | Organization directory management | The system shall allow administrators to maintain youth organizations and users to join or leave organizations. |
| FR-06 | Unified record viewing | The system shall provide detailed record pages for programs and events with date, time, location, and official source references. |
| FR-07 | Program and event registration | The system shall allow authenticated users to register for programs and events and cancel when needed. |
| FR-08 | Participation history | The system shall allow users to review joined programs, registered events, and organization memberships in their profile. |
| FR-09 | Transparency document publication | The system shall allow administrators to upload and publish disclosure documents with metadata such as type, year, quarter, barangay, and office. |
| FR-10 | Transparency board monitoring | The system shall allow encoding and display of quarterly compliance board data and monthly compliance records. |
| FR-11 | Financial disclosure view | The system shall present barangay-level financial summaries and utilization indicators to public users. |
| FR-12 | Barangay map data | The system shall present barangay-based governance information with map-linked location data. |
| FR-13 | Citizen desk support | The system shall allow users to submit citizen tickets and track their status using a reference number and email address. |
| FR-14 | Service advisories | The system shall publish operational notices and public service reminders. |
| FR-15 | Registration monitoring | The system shall allow administrators to monitor local program and event registration records. |
| FR-16 | User and role administration | The system shall allow authorized admins to manage user profiles and role assignments. |
| FR-17 | Audit logging | The system shall automatically log create, update, and delete operations in admin-managed tables. |
| FR-18 | Export and reporting support | The system shall support CSV export or equivalent record extraction where implemented in the admin interface. |

## Planned Future Feature Requirements

The following requirements were identified as desirable future extensions of LYDO Connect. These planned features are included in the overall system view and are also modeled in additional use case diagrams and reports because they support long-term operational improvement, but they are still distinguished from the currently documented core implementation scope.

| ID | Planned Requirement | Description |
|---|---|---|
| PFR-01 | Analytics dashboard | The system should provide an analytics dashboard for administrators and staff to view summary indicators, participation counts, service request trends, and operational monitoring data. |
| PFR-02 | Request and feedback management | The system should allow administrators and staff to review, categorize, assign, and resolve citizen requests or feedback submitted through the public portal. |
| PFR-03 | Notifications and alerts | The system should support in-app, email, or similar notification mechanisms for registration updates, ticket status changes, announcements, and administrative reminders. |

## Non-Functional Requirements

| ID | Requirement | Description |
|---|---|---|
| NFR-01 | Usability | The interface shall be understandable to youth users, citizens, and administrators with minimal training. |
| NFR-02 | Responsiveness | The system shall function on desktop and mobile browsers through responsive web design. |
| NFR-03 | Performance efficiency | Typical page loads and common fetch operations should complete within acceptable time under normal internet conditions. |
| NFR-04 | Reliability | The system shall preserve records consistently during create, update, registration, and tracking operations. |
| NFR-05 | Security | The system shall protect accounts and records through authentication, authorization, and controlled access to sensitive operations. |
| NFR-06 | Data integrity | Input validation, constraints, and status checks shall prevent invalid or inconsistent records. |
| NFR-07 | Compatibility | The system shall support current versions of common web browsers and typical mobile devices. |
| NFR-08 | Maintainability | The solution shall remain modular and easy to update through separated frontend, database, and deployment components. |
| NFR-09 | Portability | The system shall support user-only, admin-only, or combined deployment surfaces as configured. |
| NFR-10 | Transparency and traceability | Important administrative changes shall be recorded in audit logs for accountability. |
| NFR-11 | Recoverability | The platform shall support backup, restore, and correction mechanisms for important records. |
| NFR-12 | Scalability | The database and web architecture shall support additional records, users, and future modules without full redesign. |

## Requirement Summary

The requirement analysis shows that LYDO Connect must function as an integrated youth governance platform rather than a single-purpose website. Because of that, the methodology should retain this section and align all succeeding diagrams, schema descriptions, testing procedures, and evaluation activities with the combined public-facing and administrative workflows of the system. It is also appropriate to distinguish current core requirements from planned future extensions so that the manuscript remains complete without overstating what is already implemented.
