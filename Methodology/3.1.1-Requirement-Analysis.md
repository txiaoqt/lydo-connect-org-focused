# 3.1.1 Requirement Analysis

Requirement analysis identifies what LYDO Connect must do, who will use it, and what conditions the system must satisfy. In the current application, the scope centers on authenticated access, policy agreement gating, organization profile management, compliance document submission, budget request processing, liquidation reporting after budget release, YPOP submission and validation, barangay-level budget allocation monitoring, news and portal update management, notifications, template and document requirement management, and administrative review.

## Stakeholders and Their Needs

| Stakeholder | Primary Need | System Response |
|---|---|---|
| Organization users | Register and maintain organization records, comply with document requirements, submit budget and liquidation records, manage YPOP entries, and monitor updates | User portal with authentication, policy agreement gating, organization profile, compliance submission, budget request, liquidation reporting, YPOP workflow, notifications, and published update viewing |
| LYDO administrators | Review and validate organization records, monitor financial workflows, manage YPOP and registry records, and maintain published portal content and templates | Admin portal with overview cards, registration and compliance review, budget and liquidation monitoring, barangay allocation summaries, YPOP and YORP management, content and template maintenance, notifications, and activity logs |
| LYDO management and decision-makers | Monitor operational status, released budget summaries, and organizational compliance progress for planning and oversight | Dashboard summaries, filtered allocation monitoring, status-based workflow tracking, and audit-ready records |
| System evaluators | Check usability, reliability, and traceability | Logged actions, consistent status records, and clear workflow screens |

## Functional Requirements

| ID | Requirement | Description |
|---|---|---|
| FR-01 | User authentication | The system shall allow authorized users to sign in and sign out according to their assigned role. |
| FR-02 | Policy agreement | The system shall require non-admin users to accept the active Terms of Service and Privacy Policy before protected workflows continue. |
| FR-03 | Organization profile management | The system shall allow organization users to create and update their organization profile. |
| FR-04 | Compliance document submission | The system shall allow organization users to upload, preview, and submit required compliance documents. |
| FR-05 | Budget request CRUD | The system shall allow organization users to create, edit, update, delete, save draft, and submit budget requests. |
| FR-06 | Budget request attachments | The system shall allow budget requests to include a detailed supporting file upload. |
| FR-07 | Liquidation access control | The system shall make liquidation available only after a related budget request is released or approved for the next workflow stage. |
| FR-08 | Liquidation file upload | The system shall allow post-activity documents such as photos and attendance sheets to be uploaded to liquidation records. |
| FR-09 | YPOP workflow | The system shall allow organization users to create, update, submit, and monitor YPOP entries with supporting proof files. |
| FR-10 | News and portal updates | The system shall allow organization users to browse published news releases and portal updates relevant to their transactions. |
| FR-11 | Content and template management | The system shall allow administrators to create, update, publish, hide, or remove news, portal content, and template-related records. |
| FR-12 | Notifications | The system shall record and display workflow-related notifications to organization users and administrators. |
| FR-13 | Admin review and validation | The system shall allow admins to review organization registrations, compliance submissions, budget requests, liquidation reports, and YPOP-related records. |
| FR-14 | YORP registry management | The system shall allow admins to maintain YORP registry records associated with organization monitoring. |
| FR-15 | Audit logging | The system shall preserve important administrative actions in activity logs. |
| FR-16 | Barangay budget monitoring | The system shall group released budget amounts by barangay and district for administrative monitoring. |

## Non-Functional Requirements

| ID | Requirement | Description |
|---|---|---|
| NFR-01 | Usability | The interface shall be understandable to youth organization users and LYDO staff. |
| NFR-02 | Responsiveness | The system shall work on desktop and mobile browsers through responsive design. |
| NFR-03 | Reliability | The system shall preserve records consistently during create, update, delete, and review operations. |
| NFR-04 | Security | The system shall protect accounts and records through authenticated and role-based access. |
| NFR-05 | Data integrity | Validation rules shall prevent missing or inconsistent workflow data. |
| NFR-06 | Maintainability | The solution shall remain modular so that user and admin workflows can be updated independently. |
| NFR-07 | Traceability | Important administrative changes shall be traceable through logs and status records. |
| NFR-08 | Filtering and summary performance | The budget monitoring view shall support barangay and district filters without slowing the admin workflow. |

## Requirement Summary

The requirement analysis shows that LYDO Connect functions as a focused organization registration, compliance, and administrative monitoring platform. The current workflow is centered on authenticated access, policy acceptance, organization records, compliance document handling, budget and liquidation processing, YPOP and YORP-related monitoring, barangay-level budget tracking, news and portal update publishing, notifications, template management, and administrative review. Since the site is intended for organization registration and internal LYDO operations, public visitor features are excluded from the scope.
