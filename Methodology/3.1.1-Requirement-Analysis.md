# 3.1.1 Requirement Analysis

Requirement analysis identifies what LYDO Connect must do, who will use it, and what quality conditions the system must satisfy. For this study, requirement analysis is necessary because the system is not a simple event portal; it combines youth engagement services, transparency publication, financial disclosure, citizen service, policy agreement handling, registration synchronization, and administrative governance in one web platform. In this updated scope, the organization module provides verified Pasig-based youth organization profiles, source-backed references, related projects or initiatives, and credibility notes; it no longer provides user enrollment or join actions.

## Stakeholders and Their Needs

| Stakeholder | Primary Need | System Response |
|---|---|---|
| Youth users | Accessible information on programs, events, verified Pasig-based organizations, and participation services | Public portal with browse, registration, profile, participation history, policy agreement, and organization information modules |
| General citizens | Easy access to transparency records, financial data, and concern reporting | Transparency, citizen desk, and financial disclosure modules |
| LYDO administrators | Centralized management of youth records, transparency content, user access, and registration records | Admin portal with CRUD, dashboard summaries, registrations, compliance, documents, users, roles, and audit logs |
| SK and staff personnel | Operational monitoring, ticket handling, and data updates | Registration monitoring, barangay data maintenance, citizen desk handling, and workflow dashboards |
| Evaluators and auditors | Traceable and accountable system activity | Audit logs, policy acceptance records, role-based access, validation rules, and standardized records |

## Functional Requirements

| ID | Requirement | Description |
|---|---|---|
| FR-01 | User account management | The system shall allow users to sign up, sign in, and sign out using authenticated accounts. |
| FR-02 | Role-based access control | The system shall separate public/user access from admin access based on assigned roles. |
| FR-03 | Policy agreement enforcement | The system shall require authenticated non-admin users to accept the active Terms of Service and Privacy Policy before continuing to protected user features. |
| FR-04 | Program management | The system shall allow administrators to create, update, publish, archive, map, and configure registration settings for youth programs. |
| FR-05 | Event management | The system shall allow administrators to create, update, publish, monitor, map, and configure registration settings for youth events. |
| FR-06 | Organization directory management | The system shall allow administrators to maintain verified Pasig-based youth organization profiles with source-backed references, related project records, initiative context, and credibility notes. |
| FR-07 | Unified record viewing | The system shall provide detailed record pages for programs, events, and organizations with date, time, location, and official source references. |
| FR-08 | Program and event registration | The system shall allow authenticated users to register for programs and events, validate Philippine mobile numbers and required fields, and cancel when allowed. |
| FR-09 | External registration sync support | The system shall queue eligible portal registration records for Google Forms/Sheets sync when external attendance integration is enabled for the selected program or event. |
| FR-10 | Participation history | The system shall allow users to review joined programs and registered events in their profile. |
| FR-11 | Organization detail access | The system shall provide a dedicated "View Organization Info" action that opens complete organization details, including category, overview, mission or objectives, projects or initiatives, coverage area, beneficiaries, contact details when available, related activity year when available, and credibility notes when needed, without user enrollment or join behavior. |
| FR-12 | Source credibility and traceability | The system shall store and display source links or official references for organization records and shall allow null values for fields that cannot be verified from credible government or official pages. |
| FR-13 | Transparency document publication | The system shall allow administrators to upload and publish disclosure documents with metadata such as type, year, quarter, barangay, office, storage path, and public URL. |
| FR-14 | Transparency board monitoring | The system shall allow encoding and display of quarterly compliance board data and monthly compliance records. |
| FR-15 | Financial disclosure view | The system shall present barangay-level financial summaries, youth metrics, compliance status, and utilization indicators to public users. |
| FR-16 | Barangay map data | The system shall present barangay-based governance information with map-linked location data. |
| FR-17 | Citizen desk support | The system shall allow users to submit citizen tickets, view account-linked ticket history, and track ticket status using a reference number and email address. |
| FR-18 | Registration monitoring | The system shall allow administrators to monitor local program and event registration records, filter records, export CSV files, preview external attendance sheets, retry failed sync rows, and trigger a one-time sync run when authorized. |
| FR-19 | Dashboard and analytics summaries | The system shall provide an admin dashboard for summary indicators, participation counts, ticket trends, and operational monitoring. |
| FR-20 | User and role administration | The system shall allow authorized admins to manage user profiles, barangay assignment, and role assignments. |
| FR-21 | Audit logging | The system shall automatically log create, update, and delete operations in admin-managed tables. |
| FR-22 | Export and reporting support | The system shall support CSV export or equivalent record extraction where implemented in the admin interface. |

## Planned Future Feature Requirements

The following requirements were identified as desirable future extensions of LYDO Connect. They support long-term operational improvement but are distinguished from the currently implemented public portal, admin portal, registration monitoring, dashboard, and citizen desk workflows.

| ID | Planned Requirement | Description |
|---|---|---|
| PFR-01 | Automated notifications and alerts | The system should support in-app, email, SMS, or similar notification mechanisms for registration updates, ticket status changes, announcements, and administrative reminders. |
| PFR-02 | Advanced analytics and report scheduling | The system should generate scheduled reports, longitudinal trend dashboards, and office-ready analytics exports beyond the current dashboard summaries. |
| PFR-03 | Dedicated policy-management workspace | The system should provide an admin interface for managing policy versions instead of relying only on database migrations for active policy content. |

## Non-Functional Requirements

| ID | Requirement | Description |
|---|---|---|
| NFR-01 | Usability | The interface shall be understandable to youth users, citizens, and administrators with minimal training. |
| NFR-02 | Responsiveness | The system shall function on desktop and mobile browsers through responsive web design. |
| NFR-03 | Performance efficiency | Typical page loads and common fetch operations should complete within acceptable time under normal internet conditions. |
| NFR-04 | Reliability | The system shall preserve records consistently during create, update, registration, and tracking operations. |
| NFR-05 | Security | The system shall protect accounts and records through authentication, authorization, and controlled access to sensitive operations. |
| NFR-06 | Data integrity | Input validation, constraints, and status checks shall prevent invalid or inconsistent records. |
| NFR-07 | Compatibility | The system shall support current versions of common web browsers and typical mobile devices. |
| NFR-08 | Maintainability | The solution shall remain modular and easy to update through separated frontend, database, and deployment components. |
| NFR-09 | Portability | The system shall support user-only, admin-only, or combined deployment surfaces as configured. |
| NFR-10 | Transparency and traceability | Important administrative changes shall be recorded in audit logs for accountability. |
| NFR-11 | Recoverability | The platform shall support backup, restore, and correction mechanisms for important records. |
| NFR-12 | Scalability | The database and web architecture shall support additional records, users, and future modules without full redesign. |
| NFR-13 | Information credibility | Organization records shall prioritize official Pasig government, official youth office, official SK or barangay, and other verified public government sources to maintain trustworthiness and source-backed traceability. |
| NFR-14 | Privacy and consent | The system shall record policy acceptance for authenticated users and present current terms and privacy content before protected user workflows continue. |
| NFR-15 | Integration resilience | External registration sync shall expose pending, synced, failed, and skipped states so administrators can retry or diagnose integration issues without corrupting local registration records. |

## Requirement Summary

The requirement analysis shows that LYDO Connect must function as an integrated youth governance platform rather than a single-purpose website. Because of that, the methodology should retain this section and align all succeeding diagrams, schema descriptions, testing procedures, and evaluation activities with the combined public-facing and administrative workflows of the system. It is also appropriate to distinguish current core requirements from planned future extensions so that the manuscript remains complete without overstating what is already implemented.
