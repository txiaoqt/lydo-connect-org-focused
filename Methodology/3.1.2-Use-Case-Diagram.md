# 3.1.2 Use Case Diagram

This section presents the use case diagram for the current LYDO Connect workflow. The diagram now reflects the actual live system surface with three actors: **Public Visitor**, **Organization User**, and **LYDO Admin**.

## Figure 1. General Use Case Diagram of LYDO Connect

![Figure 1. General Use Case Diagram of LYDO Connect](./diagrams/use-case-overall-lydo-connect.svg)

## Diagram Notes

- The **Public Visitor** covers the unauthenticated web surface: public pages, account registration, password recovery, and direct viewing of individual news release records.
- The **Organization User** covers the protected organization workflow: policy acceptance, dashboard access, profile completion, document submission, budget request processing, liquidation reporting, YPOP submission, notifications, and protected news browsing.
- The **LYDO Admin** covers the administrative workflow: overview dashboard, registration and document review, YORP registry maintenance, budget and liquidation review, budget tracking with barangay allocation monitoring, YPOP validation, content maintenance, template maintenance, and system audit views.
- Include relationships represent required supporting actions such as authentication, policy loading, form validation, file validation, and status updates that generate notifications.
- Extend relationships represent optional preview actions such as opening a source post, previewing a stored file, or reviewing supporting records inside the workflow.

## Detailed Use Cases

- Public access includes browsing the landing pages, account creation, sign-in, password reset, and opening a published news release record page.
- Authentication and policy agreement remain the entry point for all protected organization workflows.
- Organization compliance now includes profile management, document submission and resubmission, budget request preparation, liquidation reporting, YPOP proof submission, notifications, and protected content access.
- Admin operations now include overview monitoring, registration review, YORP registry management, budget and liquidation decisions, barangay allocation reporting, YPOP validation management, news release CRUD, template maintenance, notifications, and activity logs.
- The diagram intentionally treats shared technical actions such as validation, file checks, previews, and notification generation as supporting use cases instead of separate actor-facing modules.

## Scope Note

Legacy transparency-post administration from earlier drafts is intentionally excluded because the current admin flow centers on budget tracking, YPOP, YORP registry, and registration review instead. Routes that only redirect to another page or do not expose a distinct rendered workflow are also not treated as separate use cases.
