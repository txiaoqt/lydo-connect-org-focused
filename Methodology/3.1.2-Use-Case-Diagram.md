# 3.1.2 Use Case Diagram

This section presents the use case diagram for the current LYDO Connect workflow. The diagram focuses on the two primary actors who perform the core operations of the system: **Organization User** and **LYDO Admin**.

## Figure 1. General Use Case Diagram of LYDO Connect

![Figure 1. General Use Case Diagram of LYDO Connect](./diagrams/use-case-overall-lydo-connect.svg)

## Diagram Notes

- The **Organization User** performs the core organization-side operations: login, profile management, compliance document submission, budget request processing, liquidation reporting, and YPOP entry management.
- The **LYDO Admin** performs the core administrative operations: registration and document review, budget and liquidation review, budget monitoring, YPOP and YORP management, content and template maintenance, and audit-oriented monitoring through notifications and activity logs.
- The blue use cases represent the main actor-facing operations of the system.
- The white use cases represent supporting actions such as authentication, policy loading, validation, status updates, notification generation, and preview operations.
- `<<include>>` relationships represent required supporting actions, while `<<extend>>` relationships represent optional or conditional supporting behavior.

## Detailed Use Cases

- Login is the common entry point for both actors and includes user authentication.
- Policy loading applies as a conditional extension for organization users before entering protected features.
- Organization workflows depend on profile and file validation before records can be saved or submitted.
- Admin workflows depend on status updates, generated notifications, and optional preview actions while records are being reviewed.
- The diagram emphasizes the current operational modules rather than every public-facing route in the site.

## Scope Note

The general use case diagram is limited to the core authenticated operations of LYDO Connect. Public informational pages and other non-core routes are not treated as separate use cases in this figure. Legacy transparency-post administration is also excluded because it no longer represents the central current workflow.
