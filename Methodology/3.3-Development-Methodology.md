# 3.3 Development Methodology

The development methodology for LYDO Connect uses a **hybrid Design Science Research (DSR) and Rapid Application Development (RAD)** approach that is aligned only with the current workflow in the repository.

## Why DSR Is Appropriate

DSR is appropriate because the study produces a working software artifact that solves a real operational problem. LYDO Connect is not only described as a concept; it is implemented as a usable web application for sign-up, sign-in, sign-out, policy agreement gating, organization profile handling, document submission, budget processing, liquidation reporting, news publishing, transparency posting, required document type management, notifications, and admin review.

## Why RAD Is Appropriate

RAD is appropriate because the application benefits from short design-build-test cycles. The workflow has evolved through repeated refinement of forms, previews, role-based routing, Supabase-backed data handling, and mobile responsiveness.

## Hybrid Use in This Study

In this study, DSR provides the research structure, while RAD provides the practical development strategy. The study identifies the problem, builds the artifact, verifies the current workflow, and then revises the application based on feedback and testing. This includes the budget monitoring tab, barangay allocation filters, and the existing approval and notification workflow.

## Process Flow Used in the Project

The project followed a structured but iterative process flow based on the current LYDO Connect workflow. It began with problem and requirement definition, where the needs of youth-organization compliance, document handling, budget accountability, transparency publishing, and administrative monitoring were identified from the LYDO operational context. This was followed by rapid prototype construction, where the user portal and admin portal were developed and refined to validate authentication, policy agreement gating, routing, forms, and record-handling behavior. Next, schema and access-control iteration took place through incremental updates to the Supabase-backed data model, including tables, relationships, status fields, constraints, storage links, and role-restricted access behavior needed by the live workflow. The process then moved to module refinement and integration, where organization profiles, document submission and review, budget requests, liquidation reporting, YPOP and YORP monitoring, transparency and news publishing, notifications, and activity-log-based admin tracking were connected and improved. Finally, testing and evaluation preparation were carried out by checking the implemented workflows against the study's quality criteria and evaluation instruments to ensure readiness for system testing and acceptability evaluation.

## Appropriateness Note

The current project does not rely on a waterfall-style process. The implementation is iterative and module-based, which is why the DSR + RAD combination is a better fit for the current app workflow.
